### 2. Расчёт количества рабочих нод (Worker Nodes)

Для обеспечения отказоустойчивости минимальное количество нод — 3. Рассмотрим два варианта.

### Вариант А: 3 ноды (минимальный HA кластер)

**Расчёт:**

1.  **Средняя нагрузка на ноду в нормальном режиме:**
    *   RAM: 30.25 ГБ / 3 = ~10 ГБ
    *   CPU: 17 ядер / 3 = ~5.7 ядер

2.  **Нагрузка при отказе 1 ноды (N-1):**
    *   Оставшиеся 2 ноды должны выдержать всю нагрузку.
    *   **Необходимо ресурсов на 1 ноду для подов:** 50% от общей нагрузки.
    *   RAM: 30.25 ГБ / 2 = **15.125 ГБ**
    *   CPU: 17 ядер / 2 = **8.5 ядер**

3.  **Добавляем служебный overhead (ОС, kubelet, DaemonSets):** +20%
    *   RAM: 15.125 ГБ * 1.2 = **18.15 ГБ**
    *   CPU: 8.5 ядер * 1.2 = **10.2 ядра**

**Итог для варианта А:**
*   Количество нод: **3**
*   Конфигурация: **8-12 vCPU, 20-24 GB RAM**
*   *Комментарий:* Конфигурация получается требовательной к CPU из-за необходимости выдерживать отказ ноды. В мирное время ресурсы будут недоиспользоваться.

### Вариант Б: 4 ноды (рекомендуемый)

**Расчёт:**

1.  **Средняя нагрузка на ноду в нормальном режиме:**
    *   RAM: 30.25 ГБ / 4 = ~7.56 ГБ
    *   CPU: 17 ядер / 4 = ~4.25 ядер

2.  **Нагрузка при отказе 1 ноды (N-1):**
    *   Оставшиеся 3 ноды должны выдержать всю нагрузку.
    *   **Необходимо ресурсов на 1 ноду для подов:** ~33% от общей нагрузки.
    *   RAM: 30.25 ГБ / 3 = **10.08 ГБ**
    *   CPU: 17 ядер / 3 = **5.67 ядер**

3.  **Добавляем служебный overhead (ОС, kubelet, DaemonSets):** +30% (для большего запаса)
    *   RAM: 10.08 ГБ * 1.3 = **13.1 ГБ**
    *   CPU: 5.67 ядер * 1.3 = **7.37 ядер**

**Итог для варианта Б:**
*   Количество нод: **4**
*   Конфигурация: **8 vCPU, 16 GB RAM**
*   *Комментарий:* Это более сбалансированное решение. Обеспечивает хороший запас прочности при отказе ноды и позволяет эффективнее использовать ресурсы в нормальном режиме.

---

### 3. Master Nodes (Control Plane)

Для отказоустойчивости control plane требуется нечётное количество нод. Для проекта такого размера достаточно 3 мастер-ноды с минимальной конфигурацией.

**Рекомендация для Master Nodes:**
*   Количество: **3 ноды**
*   Конфигурация: **2 vCPU, 4 GB RAM**

---

### 4. Итоговая рекомендуемая конфигурация кластера

На основе расчётов, для обеспечения отказоустойчивости и производительности проекта рекомендуется следующая конфигурация:

| Тип ноды        | Количество | vCPU | RAM  | Примечание                             |
|-----------------|------------|------|------|----------------------------------------|
| **Master Nodes**| 3          | 2    | 4 ГБ | Для управления кластером (etcd, API)   |
| **Worker Nodes**| 4          | 8    | 16 ГБ| Для запуска приложений                 |

### Обоснование выбора Worker Nodes (4 x 8vCPU, 16GB RAM):

*   **Отказоустойчивость:** При выходе из строя одной рабочей ноды оставшиеся три имеют достаточный запас ресурсов (`~10 ГБ RAM` и `~5.7 ядер CPU` требуется для перераспределения подов против доступных `16 ГБ RAM` и `8 ядер CPU` на ноде с учётом overhead).
*   **Запас ресурсов:** 16 ГБ RAM на ноду с запасом покрывает рассчитанные 13.1 ГБ, необходимые для работы подов и системных компонентов в аварийном режиме. 8 vCPU покрывают требование в 7.37 ядер.
*   **Баланс:** Такая конфигурация позволяет избежать как избыточности слишком крупных нод (как в варианте с 3 нодами), так и недостатка ресурсов.
